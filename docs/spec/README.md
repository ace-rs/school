# Specs

Forward-looking design specifications: feature specs, RFCs, protocol /
interface contracts, anything describing what the project intends to build
or how a component is meant to behave.

If it's a ruling on a question that came up, that's a decision — use
`../decisions/`. If it's research, exploration, or a draft, use `../notes/`.

## Format

One file per spec: `<slug>.md` (no date prefix — specs are about the thing,
not the moment). Add a status header (`draft`, `accepted`, `superseded`,
`implemented`) so readers can tell at a glance whether the spec still
describes current intent.
