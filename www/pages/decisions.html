Why it's shaped this way

The rulings behind the structure, newest insight first. Each is a frozen decision — superseded by a later one, never quietly edited. The dated originals live under docs/decisions/.

Skills get no alias, and no stub-skill workaround

2026-06-18

A skill's only invocation handle is its directory identity. The source matches only on path/basename — the frontmatter name: is display-only — so an alias: field would be silently ignored. Rather than fake a second name with a thin stub skill that forwards to the real one (banned indirection: two directories, double the maintenance, a redirect hiding the real skill), evocative second names live purely as description trigger words. A genuine second invocation earns its own real skill, or nothing.

ace-init: batched, explicit, self-contained onboarding

2026-06-13

Onboarding runs in two phases. Phase 1 skims the repo and collects every proposed change — instructions file, skills selection, docs decision — into one plan applied in a single approved batch, instead of ACE's usual per-write gate turning onboarding into a start-stop slog. Phase 2 (optional, gated) is a spec run that distills existing code into explainers. Three deliberate refusals: no dependency on a harness-native /init (it ties the skill to one harness and re-runs the heavy scan), no auto-fire (would collide with /ace's session-start orientation), and only loose coupling to the ace learn CLI.

Docs taxonomy: one gate, forced by construction

2026-07-09 · supersedes the 2026-06-07 two-axis model

Five folders routed on a single axis — the question an artifact answers — filed by a first-match gate rather than after-the-fact judgment: guides/ (how-to, use or operate), vendor/ (third-party reference), spec/ (our design + surface), decisions/ (dated rulings), scratch/ (residual exploration). The earlier two-axis model double-counted — permanence never varies within a folder, so it's a property, not a sorting axis — which let everything drain to the soft default. The fix is by construction: predicate folder names and a toll on scratch/ (each file names why it isn't spec or a decision), so nothing lands there by default. Still rejected: a human/ vs llm/ split (duplicates and rots) and llms.txt (a curated markdown index already serves it). See Architecture for the resulting shape.

Polish commits stay separate from the feature commit

2026-04-21 · PR #33

Post-audit fixes never get --amend-ed into the commit they polish. Amending hides what originally landed versus what review corrected — future readers can't tell "feature as designed" from "fix we caught." The review trail is only useful when those stay separately addressable. The rule lives in general-coding (cross-project scope); the workflow and audit skills defer to it rather than restating it.

Prefer /clear over /compact

2026-03-26

Always ace-save then /clear; never /compact. Compaction spends significant tokens on a lossy summary that's redundant once state is persisted to durable storage, where /clear gives a full reset at zero token cost. Hook-based enforcement was investigated and rejected — it's Claude Code-only and can't block the compact — so portable doc-level guidance wins.

2026-03-26

The old recall → go → compact → recall cycle loaded ~228 lines of mostly-duplicated skill text per turn. Collapsing to two skills plus a shared workflow.md (loaded once) cut that ~63%, eliminated forced per-task compaction, and put the shared rules — storage cascade, edit protocol, self-audit, commit discipline — in one place.
